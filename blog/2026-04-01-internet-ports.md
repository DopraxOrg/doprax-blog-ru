# Руководство по интернет-портам: что это такое, зачем они нужны и какие вы реально будете использовать

Порты — это, по сути, регулировщики интернет-трафика. Они невидимы, незаменимы и определяют работу каждого приложения, сервиса и соединения. Если IP-адрес — это почтовый адрес здания, то порт — это номер конкретной квартиры или офиса внутри него. Без портов ваш браузер не знал бы, куда направить входящий пакет — почтовому клиенту, веб-серверу или SSH-сессии.

Давайте разберёмся с портами по-настоящему.

# Как работают порты (за 30 секунд)

Каждое сетевое соединение использует порт-источник (случайно выбранный вашим устройством) и порт назначения (закреплённый за сервисом). Порты — это 16-битные целые числа от 0 до 65535.

* 0–1023: Общеизвестные порты — стандартизированные, зарезервированные для основных протоколов.
* 1024–49151: Зарегистрированные порты — назначены IANA для конкретных приложений и сервисов (например, Docker, Redis, Prometheus).
* 49152–65535: Динамические/частные порты — временно используются клиентами (например, ваш браузер открывает порт 54217 для загрузки веб-страницы).

⚠️ Примечание: Номер порта сам по себе не гарантирует безопасность или функциональность. Порт — это просто число. Поведение определяется тем, что на нём слушает и как это настроено.

# Самые важные порты (и что они на самом деле делают)

| Порт | Протокол | Назначение | На практике |
| --- | --- | --- | --- |
| 22 | TCP | SSH | Универсальный вход на серверы Linux. Используется для удалённой оболочки, SFTP, Git через SSH и безопасной автоматизации. По-прежнему самый критичный порт для инфраструктурных задач. |
| 80 | TCP | HTTP | Незашифрованный веб-трафик. Сегодня редко используется напрямую в продакшене — в основном перенаправляет на HTTPS или обслуживает проверки состояния. |
| 443 | TCP | HTTPS | Зашифрованный веб-трафик. Обеспечивает работу каждого современного сайта, API, панели управления и авторизации в SaaS. Де-факто стандарт для всего, что взаимодействует с пользователем. |
| 25 | TCP | SMTP (исходящая почта) | Устаревшая ретрансляция электронной почты. Заблокирован у большинства облачных провайдеров (включая Hetzner, DigitalOcean, AWS) для предотвращения спама. Не для отправки из приложений — используйте аутентифицированный SMTP через порт 587 или сервис транзакционных писем. |
| 465 / 587 | TCP | SMTPS / Submission | Безопасная отправка электронной почты. 465 устарел, но широко поддерживается; 587 — стандартный порт IETF для аутентифицированной, зашифрованной отправки. Часто заблокирован по умолчанию — проверяйте файрвол вашего провайдера. |
| 3306 | TCP | MySQL | Порт по умолчанию для доступа к базе данных MySQL. Никогда не открывайте публично — всегда ограничивайте файрволом или VPC-пирингом. |
| 5432 | TCP | PostgreSQL | То же самое — стандартный, но никогда публичный. Используйте частные сети или SSH-туннели. |
| 6379 | TCP | Redis | Хранилище данных в памяти. Как и базы данных, держите его внутренним, если только вы не используете управляемый Redis-as-a-service с аутентификацией. |
| 8080 / 8000 / 3000 | TCP | HTTP (альтернативный) | Порты по умолчанию для разработки: npm start, python -m http.server, docker run -p 8080:80. Не для продакшена — используйте 80/443 с обратным прокси (Caddy, Nginx, Traefik). |
| 2376 / 2377 | TCP | Docker Engine / Swarm | Используется для удалённого доступа к Docker-демону. Требует TLS и строгую аутентификацию — отключайте, если удалённое управление не нужно. |

# Почему некоторые порты заблокированы (и что с этим делать)

Облачные провайдеры, особенно те, которые заботятся о гигиене общей инфраструктуры (Hetzner, OVH, Vultr), блокируют определённые порты по умолчанию:

* 25, 465, 587: Чтобы спам-ботнеты не захватывали новые виртуальные машины.
* 23 (Telnet), 110 (POP3), 143 (IMAP): Устаревшие, небезопасные, редко нужны — отключены намеренно.
* Любой порт < 1024: Часто ограничен, если вы явно не запросите повышенные привилегии (например, привязка к порту 80 без root через CAP\_NET\_BIND\_SERVICE).

# Что можно сделать:

Используйте UI или CLI файрвола вашего провайдера, чтобы открыть только нужные порты — например, разрешить 443/tcp и 22/tcp, запретить всё остальное. Предпочитайте обратные прокси (Caddy, Nginx) вместо прямого доступа к приложениям через высокие порты. Для почты: направляйте через Mailgun, SendGrid или MX-настроенный ретранслятор вашего домена — не через сырой порт 25.

# Практические советы

* Никогда не открывайте базы данных или Redis публично — даже с паролями. Используйте частные сети, VPC-пиринг или SSH-туннели.
* Используйте `nmap -sT -p 1–1000 your-server-ip` для аудита открытых портов — затем закройте всё, что вы не открывали намеренно.
* Предпочитайте `localhost:3000 + caddy` вместо `0.0.0.0:3000` — привязывайтесь к loopback, если внешний доступ не требуется.

# И последнее: порты = контракты

Номер порта — это просто соглашение, договорённость между клиентом и сервером. Его ценность определяется тем, что слушает, как это защищено и кому разрешено к нему обращаться. Поэтому прежде чем открыть порт 5432, спросите:

* Эта база данных за частной сетью?
* Требуется ли надёжная аутентификация (а не просто пароль)?
* Обновлена ли она? Ведётся ли логирование? Есть ли резервные копии?

Порты открывают двери. Хорошая инфраструктура решает, кто получит ключ, что за дверью и нужно ли вообще эту дверь открывать.
